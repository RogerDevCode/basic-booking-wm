### 2.5 CRITICAL-003 — HARDCODED_NULL_TENANT_DETECTED

**File:** `f/internal/config/index.ts`
**Issue:** exported constant `NULL_TENANT_UUID` exists and is used as a bypass for tenant context, violating RLS guarantees.

#### Fix actions
1. Remove `NULL_TENANT_UUID` from `f/internal/config/index.ts`.
2. Audit all imports and uses of `NULL_TENANT_UUID` across the repository.
3. Replace tenantless code paths with explicit tenant context flows:
   - If a routine is truly global (health check, system metadata), do not use `withTenantContext`.
   - If a routine is tenant-bound, require a validated `provider_id`/`tenantId` from the authenticated session.
4. For background processors that must iterate tenants, use a two-phase pattern:
   - Query distinct tenant IDs from a global table or from `bookings` outside tenant context.
   - For each tenant ID, call `withTenantContext(client, tenantId, ...)`.
5. Add runtime guards so invalid or missing tenant IDs return `[Error, null]` immediately.

#### Verification
- `grep -R "NULL_TENANT_UUID" f/`
- `npx vitest run` for flows that previously relied on null-tenant behavior

#### Risk
High. This is a security-critical change. Apply carefully and validate with both unit tests and any integration tests that exercise administrative workflows.
